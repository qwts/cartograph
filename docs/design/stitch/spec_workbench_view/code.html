Architecture Decision Record

article ADR-0007: Event-driven Architecture

Context

Our current synchronous API calls between microservices are leading to cascading failures during high load. We need a more resilient approach to handle state changes and notifications across the system boundary.

verified T0

Decision

We will adopt an event-driven architecture using Kafka as the primary message broker. Services will publish domain events indicating state changes, rather than directly invoking remote APIs.

  • Producers must define event schemas using Avro.
  • Topics will be partitioned by entity ID to guarantee order.
  • Consumers must be idempotent to handle at-least-once delivery.
psychology T2 (Inferred)

Implementation Example

import json
from kafka import KafkaProducer

producer = KafkaProducer(
    bootstrap_servers=['kafka:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

# Payload schema mismatch detected here (F-0045)
event = {
    "event_id": "evt_98765",
    "type": "UserCreated",
    "data": { "user_id": "u_123" }
}

producer.send('user-events', event)

Consequences

Increased complexity in monitoring and debugging. We will need distributed tracing to correlate events across services.
